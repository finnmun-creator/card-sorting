# PRD: 실시간 협업 카드소팅 웹 서비스

> **문서 유형:** Reverse-Engineered PRD (구현 완료 기준)
> **작성일:** 2026-03-20
> **버전:** 1.0
> **상태:** 구현 완료

---

## 1. Executive Summary

실시간 협업 카드소팅 웹 서비스는 미팅 중 다수의 참여자가 동시에 카드를 드래그하여 티어별로 분류할 수 있는 도구다. 기획 회의, 우선순위 정렬, 항목 평가 등 다양한 협업 상황에서 팀이 동일한 화면을 실시간으로 공유하며 의견을 시각화할 수 있다.

기존에는 화이트보드나 포스트잇을 이용하거나, 비실시간 스프레드시트로 작업해 의견 수렴에 시간이 소요됐다. 본 서비스는 URL 공유 한 번으로 어디서나 동시 편집이 가능한 경량 협업 환경을 제공한다.

**핵심 가치 제안:**
- 별도 회원가입 없이 닉네임만으로 즉시 참여
- 드래그앤드롭 기반의 직관적 티어 분류
- 결과를 PNG로 캡처해 즉시 공유 가능

---

## 2. Problem Statement

### 2.1 배경

팀이 다수의 항목(기능, 아이디어, 과제 등)을 우선순위 또는 중요도 기준으로 분류해야 하는 상황은 일상적으로 발생한다. 티어 분류(S/A/B/C 등)는 직관적이고 빠른 의사결정 포맷이지만, 현재의 툴 생태계는 이 수요를 제대로 지원하지 않는다.

### 2.2 현재의 문제

| 문제 | 영향 |
|------|------|
| 실시간 협업 미지원 | 참여자가 자신의 의견을 실시간으로 반영할 수 없어 회의 속도 저하 |
| 진입 장벽 | 가입/로그인 요구 시 참여율 감소 |
| 결과 공유 불편 | 스크린샷 수동 공유, 재현 불가 |
| 맥락 손실 | 분류 근거(설명, 태그)를 카드에 붙이기 어려움 |

### 2.3 대상 사용자

- **주 사용자:** 기획자, 프로덕트 매니저, 팀 리더 (세션 생성 및 관리)
- **보조 사용자:** 개발자, 디자이너, 팀 멤버 (세션 참여 및 카드 분류)

## 3. Goals & Success Metrics

### 3.1 제품 목표

| 목표 | 설명 |
|------|------|
| 즉각적 협업 진입 | URL 공유만으로 30초 이내 세션 합류 가능 |
| 실시간 동기화 | 카드 이동이 모든 참여자 화면에 200ms 이내 반영 |
| 맥락 보존 | 카드에 설명, 태그, 이미지를 첨부해 분류 근거 기록 |
| 결과 추출 | 세션 종료 후 PNG 1회 클릭으로 결과 저장 |

### 3.2 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 세션 생성 후 첫 참여자 합류까지 시간 | < 60초 | 세션 생성 ~ 두 번째 presence 등록 시간 |
| 실시간 이벤트 전파 지연 | < 200ms (P95) | Supabase Broadcast 응답 시간 |
| 세션 당 평균 참여자 수 | ≥ 3명 | `projects.participants[]` 배열 길이 |
| PNG 내보내기 성공률 | ≥ 95% | 내보내기 시도 대비 성공 비율 |
| 재방문율 (디바이스 토큰 기준) | ≥ 30% | localStorage 토큰 재사용 비율 |

## 4. User Stories

### US-001: 프로젝트 생성자 — 새 카드소팅 세션 시작

**As a** 팀 리더로서,
**I want to** 제목과 설명을 입력해 새 프로젝트를 만들고 팀원에게 링크를 공유하고 싶다,
**So that** 회의 중 모두가 같은 보드에서 실시간으로 카드를 분류할 수 있다.

**Acceptance Criteria:**
- [ ] 홈 화면에서 프로젝트 제목(필수)과 설명(선택)을 입력해 생성 가능하다.
- [ ] 프로젝트 생성 시 기본 티어(S/A/B/C/D/F)가 자동 생성된다.
- [ ] 생성 직후 보드 화면으로 이동하며 공유 링크를 클립보드에 복사할 수 있다.
- [ ] 비밀번호를 설정하면 링크 접근 시 비밀번호 게이트가 표시된다.

---

### US-002: 참여자 — 닉네임으로 세션 합류

**As a** 팀원으로서,
**I want to** 회원가입 없이 닉네임만 입력해 카드소팅 세션에 참여하고 싶다,
**So that** 빠르게 미팅에 합류하고 즉시 카드를 분류할 수 있다.

**Acceptance Criteria:**
- [ ] 공유된 URL 접근 시 닉네임 입력 화면이 표시된다.
- [ ] 기존 참여자 목록이 칩 형태로 표시되어 빠른 선택이 가능하다.
- [ ] 자유 텍스트로 새 닉네임 입력도 허용된다.
- [ ] 동일 디바이스 재방문 시 이전 닉네임이 자동 인식된다.
- [ ] 잠긴 프로젝트의 경우 닉네임 입력 전에 비밀번호 확인이 선행된다.

---

### US-003: 참여자 — 카드 드래그앤드롭 분류

**As a** 참여자로서,
**I want to** 카드를 드래그해 티어에 배치하고 싶다,
**So that** 각 항목에 대한 내 평가를 직관적으로 표현할 수 있다.

**Acceptance Criteria:**
- [ ] 미분류 영역과 각 티어 행 간에 카드를 드래그앤드롭으로 이동할 수 있다.
- [ ] 같은 티어 내에서도 순서를 변경할 수 있다 (삽입 기반, 스왑 아님).
- [ ] 내가 이동한 카드가 다른 참여자 화면에 200ms 이내에 반영된다.
- [ ] 드래그 중 삽입 위치를 시각적으로 표시한다.

---

### US-004: 참여자 — 카드 생성 및 상세 정보 입력

**As a** 참여자로서,
**I want to** 카드를 새로 추가하고 제목, 설명, 태그, 이미지를 입력하고 싶다,
**So that** 분류할 항목에 충분한 맥락 정보를 담을 수 있다.

**Acceptance Criteria:**
- [ ] 카드 추가 시 제목(필수)과 이미지(선택)를 입력할 수 있다.
- [ ] 기존 카드를 클릭하면 상세 편집 모달이 열린다 (제목, 설명, 태그, 이미지).
- [ ] 이미지는 Supabase Storage에 업로드되며, 업로드 완료 후 카드에 표시된다.
- [ ] 이미지 클릭 시 확대 보기가 가능하다.
- [ ] 카드 삭제 시 확인 후 삭제되며 모든 참여자 화면에서 즉시 제거된다.

---

### US-005: 프로젝트 생성자 — 티어 커스터마이즈

**As a** 프로젝트 생성자로서,
**I want to** 티어의 라벨과 색상을 변경하거나 티어를 추가/삭제하고 싶다,
**So that** 우리 팀의 분류 기준에 맞는 보드를 구성할 수 있다.

**Acceptance Criteria:**
- [ ] 각 티어의 라벨 텍스트를 인라인 편집할 수 있다.
- [ ] 네이티브 컬러 픽커로 티어 색상을 변경할 수 있다.
- [ ] 새 티어를 추가하면 보드 하단에 행이 추가된다.
- [ ] 티어 삭제 시 해당 티어의 카드는 자동으로 미분류 영역으로 이동된다.
- [ ] 티어 변경 사항은 실시간으로 모든 참여자에게 전파된다.

---

### US-006: 참여자 — 실시간 협업 현황 인지

**As a** 참여자로서,
**I want to** 현재 접속 중인 팀원이 누구인지 파악하고 싶다,
**So that** 모두가 참여한 것을 확인한 후 분류를 시작할 수 있다.

**Acceptance Criteria:**
- [ ] 헤더에 현재 온라인 참여자의 아바타(이니셜 기반)가 표시된다.
- [ ] 새 참여자 합류/이탈 시 아바타 목록이 즉시 업데이트된다.
- [ ] 본인의 닉네임을 헤더에서 클릭해 변경할 수 있다.

---

### US-007: 프로젝트 생성자 — 결과 저장 및 공유

**As a** 프로젝트 생성자로서,
**I want to** 완성된 보드를 PNG 이미지로 저장하고 싶다,
**So that** 회의 결과를 팀과 공유하거나 문서에 첨부할 수 있다.

**Acceptance Criteria:**
- [ ] "내보내기" 버튼 클릭 시 2x 해상도 PNG가 다운로드된다.
- [ ] 캡처 이미지에는 모든 티어와 카드 내용이 포함된다.
- [ ] 링크 복사 버튼으로 보드 URL이 클립보드에 복사된다.
- [ ] 카카오톡/슬랙 등 메신저에 링크 공유 시 OG 미리보기가 표시된다.

## 5. Scope

### 5.1 In-Scope (구현 완료)

| 범주 | 기능 |
|------|------|
| 프로젝트 관리 | 생성, 인라인 수정, 삭제, 비밀번호 잠금 |
| 보드 접근 | 비밀번호 게이트, 닉네임 입력, 디바이스 토큰 재인식 |
| 보드 캔버스 | 멀티 티어 행, 미분류 영역, 메모/이미지 모드 토글 |
| 카드 관리 | 추가, 상세 편집, 삭제, 이미지 업로드 |
| 드래그앤드롭 | 티어 간 이동, 티어 내 순서 변경 (삽입 기반) |
| 티어 관리 | 추가, 수정(라벨/색상), 삭제 |
| 실시간 협업 | Broadcast(카드/티어 변경 전파), Presence(온라인 사용자) |
| 내보내기/공유 | PNG 캡처 (2x), 링크 복사, OG 메타태그 |

### 5.2 Out-of-Scope (미구현/제외)

| 항목 | 이유 |
|------|------|
| 회원 인증 (이메일/소셜 로그인) | 경량 진입을 위해 익명 참여 방식 채택 |
| 카드 투표/점수 집계 | 1차 버전 범위 초과 |
| 버전 히스토리 / Undo-Redo | 구현 복잡도 대비 즉각적 필요성 낮음 |
| 모바일 앱 (iOS/Android) | 웹 우선 전략 |
| 세션 내 채팅 | 별도 협업 도구(Slack 등)와의 중복 |
| 카드 자동 AI 분류 제안 | 추후 고도화 검토 |
| 다국어 지원 | 한국어 우선 출시 |

| NFR-005 | 서비스 가용성은 월 99.5% 이상이어야 한다. | Must Have | ≥ 99.5% |
| NFR-006 | 이미지 업로드 크기 제한은 10MB 이하로 설정해야 한다. | Should Have | ≤ 10MB |
| NFR-007 | 데스크탑 기준 Chrome, Safari, Edge 최신 2버전을 지원해야 한다. | Must Have | 주요 브라우저 |
| NFR-008 | 태블릿(768px 이상) 화면에서 레이아웃이 깨지지 않아야 한다. | Should Have | ≥ 768px |
| NFR-009 | 비밀번호는 평문으로 저장하지 않아야 한다. | Must Have | 해시 저장 |
| NFR-010 | 사용자 닉네임 외 개인정보를 수집하지 않아야 한다. | Must Have | 최소 수집 원칙 |

## 7. Edge Cases & Risks

### 7.1 엣지 케이스

| 케이스 | 현상 | 처리 방식 |
|--------|------|-----------|
| 동시 드래그 충돌 | 두 참여자가 동일 카드를 동시에 이동 | Broadcast 이벤트 수신 시 마지막 이벤트 기준으로 상태 덮어쓰기 (Last-Write-Wins) |
| 오프라인 복귀 | 네트워크 끊김 후 재연결 | Supabase Realtime 재구독으로 최신 상태 재수신 |
| 티어 삭제 후 동기화 지연 | 이미 해당 티어에 드래그 중인 카드 | 카드의 `tier_id`를 `null`로 처리하여 미분류 이동 |
| 동일 닉네임 중복 참여 | 같은 닉네임 두 명이 동시 접속 | Presence 채널은 디바이스 토큰 기반 구분, 표시는 닉네임 중복 허용 |
| 이미지 업로드 실패 | 네트워크 오류 또는 크기 초과 | 오류 메시지 표시, 카드는 이미지 없이 저장 |
| 빈 보드 내보내기 | 카드가 없는 상태에서 PNG 캡처 | 빈 보드 이미지 정상 생성 |
| 세션 없는 프로젝트 접근 | DB에 세션 레코드가 없는 경우 | 신규 세션 자동 생성 후 기본 티어 세팅 |
| 비밀번호 브루트포스 | 반복 비밀번호 시도 | 클라이언트 측 검증 (서버 측 rate-limiting은 Supabase RLS 위임) |

### 7.2 리스크

| 리스크 | 확률 | 영향도 | 대응 전략 |
|--------|------|--------|-----------|
| Supabase Realtime 장애 | 낮음 | 높음 | 폴링 폴백 없음 — 서비스 상태 페이지 안내 |
| 대용량 이미지로 인한 PNG 캡처 실패 | 중간 | 중간 | html-to-image 캔버스 크기 제한 존재, 이미지 모드 OFF 후 캡처 안내 |
| localStorage 미지원 환경 | 낮음 | 낮음 | 재방문 인식 불가, 매 방문 닉네임 재입력으로 graceful degrade |
| nanoid 충돌 (공유 코드) | 매우 낮음 | 중간 | DB unique 제약으로 충돌 시 재생성 |
| Vercel 무료 플랜 함수 실행 시간 제한 | 중간 | 중간 | 장시간 API 호출 없음, 모든 실시간 처리는 Supabase 직접 연결 |
| 모바일 드래그 UX 저하 | 중간 | 중간 | @dnd-kit 터치 센서 지원, 그러나 모바일 최적화는 Out-of-Scope |
